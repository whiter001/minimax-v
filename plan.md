# 待修复 Plan

基于当前日志分析和已有修复结果，下面是还需要继续处理的优化项。按优先级排序，先解决最影响稳定性和耗时的问题。

## P0

### 1. 收敛长等待逻辑（已完成）

- 问题：`v-browser wait --load networkidle` 这类等待在日志里多次卡到 30 秒。
- 目标：改成更明确的状态等待，不把“网络完全空闲”当作主路径。
- 建议方案：
  - 优先等待具体元素、标题或 URL 变化。
  - `networkidle` 只作为兜底，而不是默认等待条件。
  - 给等待动作加硬超时上限，并在超时后直接切换回退方案。
- 验收标准：
  - 日志里不再频繁出现 30 秒级等待。
  - 页面可用状态能更快被识别。
- 当前状态：已在默认系统提示词和 refine 提示中加入浏览器等待约束，减少模型生成长时间 `networkidle` 等待的概率。

### 2. 收紧图片分析超时边界（已完成）

- 问题：`understand_image` / `screen_analyze` 仍然可能出现 90 秒级等待。
- 目标：把图片分析链路的超时和重试策略收窄到可控范围。
- 建议方案：
  - 记录每次重试的真实原因，避免无意义重复调用。
  - 给图片分析增加更小粒度的 fallback，上限次数固定。
  - 若 MCP 返回错误文本，统一按失败处理，不进入下一轮假成功。
- 验收标准：
  - 不再出现多轮长时间空耗。
  - 超时后能快速切回备用路径。
- 当前状态：已把 understand_image 默认超时收紧到 60 秒，并将 fallback 限定为仅对参数校验类错误继续重试，timeout 直接收口。

## P1

### 3. 降低日志噪音

- 问题：`TOOL_DIAG` 记录的 PATH 和命令上下文过长，真正的错误被淹没。
- 目标：提高日志密度，让异常更容易被定位。
- 建议方案：
  - 把环境详情降到 debug 级别，默认日志只保留关键信息。
  - 对超长命令、PATH 和工具输入做截断。
  - 增加每次会话的简短摘要，汇总最慢步骤和失败项。
- 验收标准：
  - 常规日志文件体积下降。
  - 定位异常时不需要翻太多无效上下文。

## P2

### 4. 进一步统一参数别名

- 问题：`image_path`、`image_source`、`path`、`question` 这类别名虽然已有兼容，但仍会增加调用路径复杂度。
- 目标：让调用侧尽量只使用一套主参数名，减少 fallback 轮次。
- 建议方案：
  - 明确主参数名，其他参数只做兼容别名。
  - 在 schema 和调用生成逻辑里统一优先级。
  - 为 screen_analyze / understand_image 增加更直接的输入规范说明。
- 验收标准：
  - 失败重试次数减少。
  - 日志里不再反复出现同一类参数转换。

## 推荐顺序

1. 先做长等待收敛。
2. 再收紧图片分析超时。
3. 最后整理日志噪音和参数别名。
